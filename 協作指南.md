# 河內＆沙壩行程 協作指南

這份文件說明「多人／多次協作」時，如何修改行程內容而不破壞網頁的呈現規格。

**核心原則：`行程資料.txt` 是唯一的編輯來源（source of truth）。`index.html` 是同步產出物，任何人都不應該直接手改 `index.html` 裡的行程資料，一律透過「編輯 txt → 請 AI 同步」的流程進行。**

文件分兩部分：
- **Part A：給人看的** — 你只需要看這部分，照著格式編輯 `行程資料.txt`
- **Part B：給 AI 看的** — AI 助理同步資料到 `index.html` 時要遵守的規則，人類協作者不需要記住這部分

---

## Part A：給人看的操作規則

### 檔案角色

| 檔案 | 用途 | 誰可以編輯 |
|---|---|---|
| `行程資料.txt` | 行程內容的唯一編輯來源 | 所有協作者 |
| `index.html` | 實際網頁，由 AI 根據 txt 同步產生 | 只由 AI 同步寫入，人不要手改行程資料部分 |

### txt 檔案格式說明

每一天是一個區塊，結構固定如下：

```
第X天｜M/D(週幾)
標題: 一句話標題
摘要: 一句話摘要
-------------------------

name: 景點名稱
mapName: Google Maps 用的英文/官方名稱
plusCode: Plus Code（在 Google Maps 上搜尋景點時可取得）
city: 所在城市（如 Hanoi / Sapa / Lao Cai）
type: 景點 / 餐廳 / 交通 / 住宿（僅限這四種）
note: 一句話簡介
address: 完整地址
detail: 詳細說明（開放時間、注意事項等，可以多行）
```

同一天內，每個景點區塊之間空一行分隔；天與天之間空兩行分隔。

### 操作規則

**新增景點**
1. 找到要新增的那一天區塊
2. 複製一組完整的 8 個欄位（name/mapName/plusCode/city/type/note/address/detail）
3. 貼在該天最後一個景點後面，空一行分隔，填入新內容
4. 每個欄位都建議填，若真的沒有可以留空或寫「無」，但 `name` 和 `type` 一定要填

**刪除景點**
1. 找到要刪除的景點區塊
2. 整段（含 8 行欄位）刪除，確保上下維持一個空行分隔即可

**修改景點**
- 直接修改對應欄位的值，欄位名稱（`name:` `type:` 等）不要改動

**新增一天**
1. 複製一整個天區塊（含 `第X天｜日期`、`標題`、`摘要`、分隔線、所有景點）
2. 修改 `第X天` 編號（第幾天）、日期、標題、摘要
3. 填入該天的景點

**刪除一天**
- 整個區塊（從 `第X天｜...` 到下一個 `第X天｜...` 之前）全部刪除
- 注意：如果刪除的不是最後一天，記得該天之後的天數編號（第X天）建議跟著往前遞補，避免跳號

**修改某天的標題／摘要／日期**
- 直接修改 `標題:` `摘要:` 或 `第X天｜M/D(週幾)` 那一行

### 同步流程

1. 編輯完 `行程資料.txt` 後存檔
2. 跟 AI 助理說「同步一下」或「幫我更新網頁」
3. AI 會讀取 txt、比對修改，只更新 `index.html` 裡的行程資料，不會動到版面/樣式
4. 建議同步後打開 `index.html` 或請 AI 做語法檢查，確認無誤後再視需要 commit

## Part B：給 AI 協作者的同步規範

當人類協作者請你「同步」或「更新網頁」時，依以下規則執行：

### 你的角色與限制

- 只能修改 `index.html` 中 `const DAYS = [...]` 這個常數的內容（[index.html](index.html) 中 `<script type="text/x-dc" ...>` 區塊內）
- **禁止**修改：樣板 HTML（`<x-dc>` 內的結構）、`THEMES`、`Component` class、`TYPE_COLOR`、任何 CSS/style，除非人類協作者明確要求調整外觀
- 這是為了保證 UI/UX 規格在多次協作後維持一致

### 解析 txt 的規則

1. 以 `第X天｜` 開頭的行作為每天的區塊起點
2. 區塊起點行同時包含 `num`（第X天）與 `shortLabel`（｜後面的日期字串）
3. 緊接著的 `標題:` 對應 `title`，`摘要:` 對應 `overview`
4. `-------------------------` 分隔線之後、下一個空兩行之前，是該天的景點清單
5. 景點區塊以空行分隔，每個區塊內用 `key: value` 解析，其中 `detail` 欄位允許多行（換行需保留為 `\n`）
6. 欄位對應到 JS 物件 key：`name / mapName / plusCode / city / type / note / address / detail`

### 資料驗證規則（同步前檢查）

- `type` 應為「景點／餐廳／交通／住宿」四者之一。若出現新分類，提醒使用者這個分類不在 `TYPE_COLOR`（[index.html:98-103](index.html#L98-L103)）裡，會套用預設灰色，除非使用者要求新增顏色
- 字串中若含單引號（如 `St. Joseph's Cathedral`），寫入 JS 時該欄位改用雙引號包住，避免破壞語法（可參考既有寫法 [index.html:110](index.html#L110)）
- `name` 與 `type` 為必填，其餘欄位若使用者留空，可寫空字串 `''`
- 新增一天時，`photo` 欄位目前樣板未實際使用（dead field），可省略或留空字串，不影響畫面
- 不需要刻意保留舊資料中 `num`/`shortLabel` 尾端多餘的 `\n` 或空白（那是原始資料的既有瑕疵，不影響顯示，但新資料不必模仿）

### 寫回 index.html 的規則

- 新增/刪除/修改景點：只調整對應那一天 `landmarks` 陣列裡的物件，用 `Edit` 工具做最小範圍的字串取代，不要整段重寫 `DAYS`
- 新增一天：在 `DAYS` 陣列對應位置插入新的天物件，維持相同的物件結構（`num, title, shortLabel, overview, photo, landmarks`）
- 刪除一天：移除整個天物件（含前後逗號處理，確保陣列語法正確）
- 修改天數編號時，若使用者要求跳號調整，同步更新前後天的 `num`

### 同步後自我檢查清單

1. 用 Node 執行語法檢查（例如把 `DAYS` 字串抽出並 `eval`，確認可解析、天數與景點數符合預期）
2. 檢查有無重複逗號、缺少逗號等常見 JS 陣列語法錯誤
3. 簡短回報這次同步變更了什麼（新增/刪除/修改了哪些天、哪些景點），讓人類協作者可以快速核對
4. 不要主動 commit / push，除非使用者明確要求
